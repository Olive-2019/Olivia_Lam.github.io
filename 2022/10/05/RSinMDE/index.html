    推荐系统在软件建模中的作用的综述

    三个问题

    1. 推荐系统在软件建模中起什么作用？
      通常用于完成和修复工件。
      通常是独立于语言的，特定的一般针对UML或过程建模符号
    2. 哪些推荐技术用于支持软件建模？评价方法？
      基于内容>基于知识
      评价方法：离线
    3. 推荐系统还有什么可做的？
      几乎没有用于模型转化或代码生成的推荐系统，很少推荐系统用于创建、重用、查找工件。此外，还有有效的软件建模的工件库、协同过滤利用群体知识、基于用户的推荐系统评估、与低代码平台的有效寄成机制

    RS in MDE 模型驱动工程中的推荐系统

    文章分了四个维度来叙述模型驱动工程中的推荐系统，分别是推荐系统在什么领域解决什么问题、提出的工具有什么特点、使用到了什么样的推荐系统以及怎么评价这个推荐系统。

    Domain 领域


    领域部分将RS能解决的问题正交地分为了三个维度，分别是artifacts，即能针对什么工件、LanguageIndependent，即是否语言独立的、Purpose，即RS针对的是什么什么目的。

    针对工件和目标，又正交地分成了24个维度，将多个方法分别分到了这些维度上。

    Complete 完成模型

    第一种是使用搜索技术扩展部分模型形成完整模型。
    使用搜索技术DIA GEN 基于超图语法规律（hyper-graph grammar rules）生成可能的工件，具有高计算复杂度，DIG MDE 发现时间复杂度过高后，会建议手动修改模型保证模型补全完成。
    第二种是给出一个模型逐步完成的建议。这种建议来自模型库中已有的模型碎片。

    Create 创建模型

    创建模型的相对较少，只有两个，一个是针对元模型的，另一个是针对模型的。
    针对元模型的是DSL-maps，用思维导图表达DSL需求，DSL-maps推荐元模型模式，用户选择以后可以再改进。
    针对模型的是用例图的建模助手（UC-check）。创建用例图时，向导会推荐合适的参与者和用例。

    Find 查找工件

    查找模型工件。从（异构）库中查找并根据用户适用性排序。

    Repair 修复工件

    修复工件。感觉跟推荐系统关系并不是很大的样子

    Reuse 复用

    不只是查找，在新的上下文中提供复用工件的帮助。

    Other purpose 其它目的

    一些比较特殊的需求，没太看懂。

    Tooling 工具

    工具部分介绍了工具的成熟度、工具的独立性、工具中推荐系统的触发方式、推荐系统的互动方式

    Maturity 成熟度

    三个级别的成熟度，没有实现的提议，有概念的原型，完整实现（包含框架、插件、MDE）。

    绝大部分提供了完整实现，下面为完整实现的28个方法中的分类。

    Tool Independence 工具独立性

    工具独立性只能在原型和完整实现中讨论，建议不在讨论范围中。根据是否可以集成到其它MDE工具中判断是否具有工具独立性。

    trigger recommendations 触发方式

    触发建议的方式：按需、主动。前者，用户需要主动触发建议；后者，系统主动提建议

    Recommendation enactment 制定建议方式

    制定建议的方式：
    手动：给出一个序列，用户决定使用方式
    交互式：推荐系统允许用户选择项目，然后应用到某个上下文中
    自动：用户不参与
    半自动：自动的过程中用户给出一点操作

    Recommendation 推荐系统

    推荐系统方面介绍了推荐方法、用户偏好收集、推荐的项目以及推荐度。

    Recommendation Method 推荐方法

    基于内容的方法使用不同内容编码和相似性来表示

    协同过滤利用了用户的历史选择信息

    基于知识的方法可以分为基于约束和基于案例的。基于约束是指寻找一组满足领域相关规则的项目，基于案例是指搜索类似案例的过往解决方案来解决。

    混合方法是指将前面三种方法混合。

    还有一些其它的，使用静态分析或搜索的方法。

    Hermes框架适用度比较广。

    User Preference 用户偏好

    获取方式包括显式和隐式。隐式是监控用户动作；显式是直接收集用户信息，如调查问卷等。绝大多数都是隐式收集用户偏好
    另外，收集的用户偏好可以是长期的也可以是短期的。90%的算法收集的都是短期偏好。

    Recommendation Item 推荐的项目

    项目可能是工件（碎片或完整工件）、建模活动中的建议、修复过程中的编辑动作。

    Recommendation Degree 推荐度

    推荐度分为两个维度，分别是基数和排序。基数指建议的总数；排序指建议是否被排序。

    排序的指标包括相似性、修复更多的错误、更少的修改、根据一个预训练的机器学习模型计算。

    Evalution 评价

    Method 方法

    Offline Experiment 离线实验

    使用过往的数据集模拟现实情况。通常面临的问题是数据集难以获取。

    数据集通常有四种来源：

    1. 生成数据
    2. 从现成的库中获取数据
    3. 从现有文章中获取实例
    4. 从公司获取现实世界的数据

    Online Experiment 在线实验

    应该就是A/B测试，两组用户分别使用和不使用RS进行测试，对结果进行分析评估。不是很懂跟用户调查之间的差别，感觉可能是更注重实时性。

    User Study 用户研究

    通常会涉及到一组用户，通过使用和不使用RS去完成一些任务再分析可行性，并通过问卷收集信息。

    用户研究通常有三种类型。

    1. 使用RS来完成任务
    2. 用户在A/B测试中使用RS
    3. 将推荐的项目和专家推荐的对比

    Metrics 指标

    Application-independent metrics 应用独立的指标

    最常用的是基于排序的precision、recall、F-Score

    Application-dependent metrics 依赖于应用的指标
