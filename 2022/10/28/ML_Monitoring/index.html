低代码开发机器学习监控系统

简化版摘要

根据2019年微软提出的ML工作流，Panagiotis Kourouklidis关注ML模型部署后的监控问题，在2020年在这个领域发了一篇survey并提出了一些研究方向，2021年提出了一个元模型试图解决自动生成ML模型监控程序的问题。同时也给出了该元模型基于Kubernete生成系统的一个例子。

其中比较有意义的就是这个元模型以及提及的一些工具。跟DWF目前的工作关系不大，但是可能对未来的工作有比较重要的意义。

引文中大多为ML领域的文章，对我们来说比较有意义的可能是微软2019年发的机器学习中的软工。引用这两篇的文章大多为review，并没有对这项工作做出什么拓展。

文章提到的对于ML流程自动化的工具对我们来说相对较为有用。

背景

工作流

部署后漂移问题

有监督学习中的X，Y通常被视为服从联合概率分布P(X, Y)的随机变量。由于真实世界情况复杂且不稳定，概率分布会发生变化，即概念漂移、采样漂移、先验概率漂移或更一般的数据集漂移。

此处主要讨论数据漂移和概念漂移两种。

数据漂移

数据漂移可能是由采样机制，环境中的未知因素随时间变化引起，数据漂移不一定导致输出错误，没有数据漂移也不一定能保证输出正确。目前可以通过检查数据输入来检测数据漂移。

概念漂移

概念漂移指输入-输出映射发生变化。概念漂移可能是真实的，也可能是虚拟的。真实的指的是由于某些未知因素发生改变，导致映射发生改变；虚拟的指的是映射实际并没有发生改变，只是由于其它原因观测到这个现象（比如有偏采样）。

实例

客服中心的服务对象的反馈：使用等待时长和服务时长拟合反馈

目前模型是总时长小于20则为positive

模型部署后可能存在四种状态，无漂移、数据漂移、概念漂移、两种漂移并存。

无漂移。理想状态，即输入的数据分布不改变，模型也一直符合现实状况

数据漂移。例如某个客服没上班，等待时长变长，但是用户并不会因此愿意等待更长时间。这种数据漂移并没有导致输出错误

概念漂移。模型不再符合现实情况，但是数据分布不发生改变，输出存在错误。如用户愿意等的更长，则现实模型划分线应该上移。
这种情况下，模型需要重新学习新数据。

两种漂移并存。有偏采样发生时，输入数据的分布自然与学习集不同了。同时，由于学习集的分布与真实输入不同，训练出来的模型自然不能用于真实输入。但是我们会以为观察到数据漂移和真实概念漂移。目前区分真实和虚假的概念漂移仍困难。

研究方向

希望开发一个抽象化的低代码的解决方案，包括DSL和参考实现。允许用户只需要配置模型监控方案，不必关注具体细节。将模型监控的任务分成三大类，分别是，数据捕捉、算法执行、响应漂移。

数据捕捉

为了检测概念漂移，需要存储模型输出并与反馈进行对比。在前面提到的实例中，可以建立反向代理服务器转发请求和响应，并将事件写到数据库中。

收集数据的两个重要来源：

  • 模型的输入输出
  • 反馈

收集反馈的困难在于收集方式取决于模型，例如推荐系统的反馈收集只需要记录一下是否点击，而图片分类器的反馈收集必须要人工介入。

算法执行

检测算法的性能是非常重要的
希望能使用DSL来描述漂移检测算法，然后生成适用于不同框架的算法

响应漂移

方法是多样的，可以包含以下两种

  • 检测到漂移时简单地发送email
  • 自动触发重新训练模型的流程

前者多用于数据漂移，后者多用于概念漂移。

实施方案技术路线

部署环境假设

  • ML模型持续接收未标记的样本（推理请求），并需要返回一个预测结果（推理响应）
  • 未标记样本可能在未知时长后接收到真实标签
  • 定期根据ML模型接收到的最新数据决定是否修正模型
  • 由领域专家定义不同种类的漂移响应策略

元模型组件

  • 指定想要监控哪一个ML模型的机制
  • 指定哪些推理请求和推理响应字段需要被捕捉。（这也是在命名被捕捉字段的方法。命名可以方便其它实体单独引用它。）
  • 定义周期性漂移检测执行机制
  • 定义检测到漂移时，需要采取的措施

元模型设计

虽然希望元模型和具体底层技术完全分离，但为尽快从算法专家处得到反馈，这个版本的元模型做了一些妥协。下文将会提及这些妥协。

deployment抽象类包括了ML监控方法的所有方面。在系统中，通常有一个Model子类的实例和任意数量的DriftDetector子类的实例。

Model类包含了名称、序列化ML模型的URL、创建它的框架名称、多个输入的IO实例和一个输出的IO实例。

IO类是模型的输入输出，包含输入/输出的名字和数据类型的枚举属性。

DriftDetector类周期性地执行漂移检测算法，属性包含名字、频率、email、定制容器。执行的算法可能是底层平台包含的算法也可能是自定义的算法。DataDriftDetector需要训练数据和现有数据。ConceptDriftDetector需要真实数据的预测标签。

元模型的妥协

  • 为了简化展示，ML模型将会被部署到web端点。因此Model类中需要序列化ML模型的URL、创建它的框架名称这两个属性，这两个属性并不需要被包含进来，因为部署过程可能是单独执行的。
  • 区分两种漂移检测器。区分是为了简化底层技术实现。
  • DriftDetector属性中包含了email，检测到模型漂移的行为只能是发送email了。（可以抽象化成一个超类）

系统生成

使用M2T转化语言，基于Kubernetes生成了一个具体的ML监控系统。这个方法的优点是没有生成过程代码。生成的组件包括两种类型，一种是开源的定制的Kubernete资源，另一种是容器注册表中注册的容器化应用。

工件

包含了Inference Service（推理服务）、Message Queue、Database Writer、Database、Proxy（代理）、Configuration和 Drift Detection Execution等组件。

Inference Service组件包括两个功能，分别是Model Server和Logger。Model Server负责响应各种应用程序发送的推理请求，Logger负责记录请求和响应事件。

Message Queue从Logger接收消息，路由至Database Writer。

Database Writer主要功能有两个，其一是根据算法专家提供的消息初始化数据库，其二是将捕获到的数据存储到数据库中，捕获的数据来源包括logger记录的事件和ML模型的性能反馈。

Database组件使用MySQL数据库。

Drift Detection Execution组件周期性执行算法专家指定的漂移检测算法。算法专家可以只实现漂移检测算法，并指定实现这个算法的容器映像，不必实现数据检索和通知发送的部分。

Configuration通过使用配置文件，容器化应用程序根据算法专家规定的需求调整自己的功能。

Proxy需要用同一个主机名接收推理请求和反馈数据，所以需要使用代理来进行转发。代理接收请求后根据请求路径转发到inference service或database writer。

平台

为了最大程度地降低算法专家的使用门槛，建立了web应用。主要功能包括创建和提交MDE模型。辅助功能包括查询、管理当前所有的ML模型、数据集，注册漂移检测算法。

对论文的优势和局限性的评价

对我们现在的工作有什么启发

这个作者做的东西有点偏离了LCDP，是自动生成的ML Model监控程序的工作。对我们目前的工作并没有什么帮助，但是假如我们要做RS嵌入LCDP平台实现智能推荐的话，对RS系统的监控也是一个值得思考的工作。

有价值的相关工具

文章里提到了一些ML流程自动化的工具。

MLflow

做元数据追踪，提高实验可复现性。很容易和现有机器学习框架结合使用，且有直观的UI。但非端到端全自动的工具，仍有数据准备和模型监控未覆盖。

Tensorflow extended

端到端全自动。但仅支持tf core，模型监控方面做得比较单调，只支持针对分类特征的数据漂移检测，且只提供一种距离度量。

alibi-detect

开源的检测库，提供数据漂移、异常值、对抗样本等检测支持。
